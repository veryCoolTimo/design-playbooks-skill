# Design Intelligence Skill — спецификация

**Дата:** 2026-07-10 · **Статус:** approved-pending-review

## Проблема

AI-агентам при фронтенд-работе хронически не хватает дизайн-референсов. Официальный
frontend-design skill даёт принципы («не используй Inter»), но не доказательную базу.
Mobbin MCP даёт 600k скринов, но платно и без синтеза. Никто не даёт **правил,
выведенных из реального корпуса** («как реально строят pricing-страницы»).

У нас есть корпус: 407 сайтов (~1150 скринов постранично: home/, pricing/, blog/…),
75 эталонных DESIGN.md (getdesign.md), 248 дизайн-работ, 93 app-иконки, 49 OG-картинок
с тегами (recent.design API).

## Продукт

**Design Intelligence Skill** — глобальный Claude Code skill + локальная библиотека знаний.

Шесть возможностей:
1. **Page playbooks** (главный продукт) — правила по типам страниц, синтезированные
   из корпуса: home, pricing, blog, about, docs, contact и др.
2. **Style playbooks** — 9 стилевых гайдов (минимал, тёмный tech, неон, playful,
   иллюстрации, крупная типографика, elegant serif, фото, градиент) с do/don't.
3. **Корпус профилей** — 400+ сайтов в формате DESIGN.md (наша генерация).
4. **DESIGN.md-генератор** — скилл генерит DESIGN.md для проекта пользователя
   («как Stripe, но зелёный») или любого URL. Бесплатная альтернатива платной
   услуге getdesign.md.
5. **Live Analyzer** — скриншот и разбор любого живого URL на лету (Playwright);
   корпус — кэш, не потолок.
6. **Verify-цикл** — после генерации UI скилл скринит результат и сравнивает
   с референсом глазами.

## Research-прогон (multi-agent Workflow)

Приоритет по ценности:

**Фаза 0 — Калибровка.** Vision-агент анализирует скрины 3–5 сайтов, у которых ЕСТЬ
эталонный DESIGN.md (stripe, figma, supabase…). Сравниваем вывод с эталоном,
итерируем промпт до совпадения по палитре/шрифтам/компонентам. Только после
этого — массовая генерация. 75 эталонов = ground truth, не редистрибутив.

**Фаза 1 — Page playbooks.** Отдельный playbook для типа страницы при ≥15 примерах
в корпусе (home ~200, pricing ~120, blog ~60, about ~45, features, solutions,
customers, careers, enterprise…); типы с <15 примерами сводятся в общий
`pages/misc.md`. Агенты смотрят выборки скринов типа → правила: структура,
паттерны, типовые секции, ошибки, лучшие примеры. Coverage-бейдж обязателен.

**Фаза 2 — Профили сайтов.** Агент на сайт: все скрины папки + meta.json/DESIGN.md
если есть → профиль по калиброванной схеме. Резюмируемость: профиль существует → скип.

**Фаза 3 — Style playbooks.** 9 агентов: профили своего стиля → гайд
(принципы, формулы палитр, do/don't, канонические примеры).

**Фаза 4 — Домены.** apps.md, icons.md, og.md + доменные playbooks
(как делать app-иконку, как делать OG-картинку).

**Фаза 5 — Catalog + валидация.** Скрипт собирает catalog.json из профилей;
проверка: все пути существуют, все slug'и покрыты.

## Скилл (SKILL.md)

- **Триггеры:** авто — любая работа над UI/фронтом (создание, правка, стиль);
  явно — `/design-refs <запрос>`.
- **Алгоритм:**
  1. Определить: тип задачи (новый дизайн / правка / генерация DESIGN.md / консультация),
     тип страницы, стиль/индустрию. При правке существующего фронта — сначала
     прочитать код проекта (tailwind config, CSS-переменные) и определить текущий стиль.
  2. Загрузить: 1 page-playbook + 1 style-playbook + 2–4 профиля (grep по catalog.json).
  3. Если медиа доступно — посмотреть 2–3 скрина; иначе — живые URL / Live Analyzer.
  4. Применить как правила: токены из профилей, do/don't из playbooks.
  5. По запросу — verify: скрин результата vs референс.
- **Совместимость:** позиционируется как evidence-layer к официальному
  frontend-design skill (он — направление, мы — доказательства). Не конфликтует.
- **Бюджет:** SKILL.md ≤150 строк; знания подтягиваются прогрессивно.

## Публикация (позже)

- Репо: скилл + catalog + profiles + playbooks + scripts. БЕЗ media/, БЕЗ 75 чужих
  DESIGN.md (остаются локальной калибровкой).
- Опция: медиа-архив в облако (R2), install-скрипт качает по желанию пользователя.
- Лицензионная позиция: наши тексты — собственный анализ; чужие ассеты не шипаем.

## Критерии успеха

1. «Сделай pricing-страницу для dev-tools в тёмном стиле» → скилл подтягивает
   pricing-playbook + dark-tech-playbook + 2–3 профиля → результат следует правилам.
2. «Поправь hero на нашем лендинге» → скилл читает код, определяет стиль,
   не ломает его, улучшает по референсам.
3. «Сгенери DESIGN.md для проекта как Linear» → валидный DESIGN.md с YAML-токенами.
4. Все пути в catalog.json/профилях резолвятся; playbooks имеют coverage-бейджи.

## Вне скоупа (сейчас)

- Облачная установка медиа (следующий этап).
- RAG-индексация в timo-rag (опция после).
- Автогенерация UI-кода компонентов (это делает 21st.dev; мы — знания, не кодоген).
